MapStructor Speed Lab

Isolated experiments for the machine that makes maps feel fast. Each module changes one part of the machine — tiling, encoding, scheduling, hardware — and puts a meter on what your eyes actually get.

Live

Raster time-encoding

The year of every railroad is baked into pixel values once. Scrubbing changes ONE number on the GPU — visibility at any date, instantly, at any data size.

 Live

Raster + vector, simultaneously

Both on one map, one slider: the raster (purple) answers instantly while the real vector tileset (orange) filters the current product way. Any orange trailing purple is the lag, made visible.

 Planned

Tiler playground

Tweak each part of the tiling algorithm — simplification, sub-pixel drop, per-zoom budget — isolate a single zoom level, regenerate, and eyeball the result immediately.

 Planned

Scrub strategy shoot-out

setFilter vs paint-expression vs GPU uniform vs raster, on the same dataset, same viewport, with live ms/frame and fps meters.

 Planned

Hardware simulator

Synthetic CPU load, device-pixel-ratio spoofing and slow-network tiles — see the map the way a mid-range laptop or phone sees it.

Serve locally (e.g. python -m http.server 8000 at the repo root) and open /test/speedlab/. Nothing here ships to the product until it wins on the meters.
