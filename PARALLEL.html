Parallel development — the two-window system

Two VSCode windows, two folders, one repository. While Claude runs in one window, you work in the other — and git does the recombining, not hand-copying.

Design doc — nothing is set up yet. Activate whenever you like: git worktree add ../mapstructor-B -b dev-B (then a serve-8001.bat in the new folder), or just ask Claude: “set up parallel dev”. Undo: git worktree remove ../mapstructor-B.

Both folders are the same git repository (a “worktree”) — one shared history, two checkouts. GitHub Desktop sees one repo; the folder you're in decides the branch. You never switch branches — you switch folders.

The rules of the road

1
One window = one folder = one port.Never open the same folder in both windows; never share a port.
2
Commit often, where the work happened.Prefix Window B commits with B: so history reads cleanly.
3
The database is shared.There is ONE Supabase. Schema changes and feature edits from either window hit both instantly. Destructive experiments → test-prefixed tables/maps.
4
Gitignored files don't travel.Secrets and tokens never copy into B automatically — copy by hand if B needs one.
5
B starts at the last commit.Uncommitted work in A doesn't exist in B. Commit in A, then in B run git merge master to catch up.

The daily loop

Open A + B in two windows→ Start each serve bat once→ Task to A's Claude→ Switch to B while it runs→ Test :8000 / :8001→ Commit each side in its window

The two ports mean separate logins, storage and service workers — the windows can't contaminate each other's browser state.

Bringing B's work into the product — “merge B”

In Window A's terminal (or ask Claude: “merge B”):

git merge dev-B
  • No overlap → done instantly. A now contains B's feature.
  • Overlapping edits (same lines of the same file, e.g. editing.js) → git pauses and marks each spot with <<<<<<</>>>>>>>. VSCode shows Accept Current / Accept Incoming buttons — pick a winner per spot, save, then git add . && git commit.
Keep future conflicts small: every day or two, in Window B run git merge master so B stays close to the product.

Disposal / renewal

  • Retire B (from A): git worktree remove ../mapstructor-B, then git branch -d dev-B once merged.
  • Fresh B for the next feature: git worktree add ../mapstructor-B -b dev-B2
  • B is disposable — merged history is permanent either way.
